# 绮管电商后台 - Code Review 检查清单

> **使用说明**: 每个 Pull Request / Merge Request 提交前，请逐项检查以下内容。所有标记为 ⚠️ 的项目为**必须通过**项。

---

## 一、功能性检查 (Functionality)

### 需求实现
- [ ] 是否完整实现了需求文档中描述的所有功能点？
- [ ] 边界条件是否已处理？（空值、null、undefined、空数组、超大数值等）
- [ ] 异常场景是否有合理的降级或错误提示？
- [ ] 功能是否存在回归风险？是否影响了已有模块？

### 接口契约
- [ ] API 响应格式是否符合项目约定（参考 `utils/responseHelper.js`）？
- [ ] 新增/修改的 API 端点是否已在 `api/index.js` 中注册？
- [ ] 请求参数校验是否完整（参考 `utils/validation.js`）？
- [ ] HTTP 状态码使用是否正确？

### 数据库操作
- [ ] 数据库 schema 变更是否附带 migration 脚本？
- [ ] SQL 查询是否使用了参数化绑定（禁止字符串拼接 WHERE 条件）？
- [ ] 事务操作是否正确处理了 commit/rollback？
- [ ] 批量操作是否有分页或限制，防止内存溢出？

---

## 二、安全性检查 (Security) ⚠️

### 注入防护
- [ ] ⚠️ **SQL 注入**: 所有动态 SQL 必须使用参数化查询（`db.prepare` / `?` 占位符）
- [ ] ⚠️ **XSS 防护**: 前端渲染用户输入时是否进行了转义？
- [ ] ⚠️ **命令注入**: 是否避免了将用户输入直接传入 `exec`、`spawn` 等

### 认证与授权
- [ ] 敏感接口是否添加了认证中间件（`middleware/auth.js`）？
- [ ] RBAC 权限校验是否正确（`middleware/rbac.js`）？
- [ ] 用户是否能越权访问他人的数据（水平越权）？
- [ ] 敏感操作是否有操作日志记录？

### 数据安全
- [ ] ⚠️ 无硬编码密钥、密码、Token（必须使用环境变量）
- [ ] 敏感数据（手机号、身份证）在日志中是否脱敏？
- [ ] 文件上传是否校验了文件类型和大小（防止恶意文件上传）？
- [ ] 环境变量变更是否同步更新了 `.env.example`？

### 输入验证
- [ ] ⚠️ 用户输入是否经过 sanitize/validation（参考 `utils/validation.js` 和 `utils/securityCheck.js`）？
- [ ] 枚举值是否做了白名单校验？
- [ ] 日期、金额等特殊格式是否做了严格校验？

---

## 三、性能检查 (Performance)

### 数据库性能
- [ ] 是否存在 N+1 查询问题？批量查询应使用 `IN (...)` 或 JOIN
- [ ] 大表查询是否添加了合适的索引？
- [ ] 分页查询是否使用了 `LIMIT/OFFSET` 或游标分页？
- [ ] 是否有不必要的 `SELECT *`？只查询需要的字段

### 前端性能
- [ ] 列表页面是否实现了虚拟滚动或分页加载？
- [ ] 大数据量渲染是否使用了懒加载（`directives/lazy-load.js`）？
- [ ] 是否有不必要的重复请求或轮询？
- [ ] 组件卸载时是否清理了定时器、事件监听器（防止内存泄漏）？

### 资源优化
- [ ] 图片/文件是否压缩或使用了 CDN？
- [ ] 打包产物体积是否有异常增长？
- [ ] 是否存在不必要的第三方依赖引入？

---

## 四、可维护性检查 (Maintainability)

### 代码清晰度
- [ ] 函数/方法职责是否单一（单一职责原则）？
- [ ] 函数长度是否合理（建议不超过 50 行）？
- [ ] 嵌套层级是否过深（建议不超过 3 层）？
- [ ] 是否消除了重复代码（DRY 原则）？

### 命名规范
- [ ] 变量/函数命名是否语义明确（见名知意）？
- [ ] 文件命名是否符合项目约定（kebab-case）？
- [ ] 常量是否使用大写蛇形命名（`UPPER_SNAKE_CASE`）？
- [ ] 组件命名是否使用 PascalCase？

### 注释与文档
- [ ] 复杂业务逻辑是否有必要的注释说明？
- [ ] 公共函数是否有 JSDoc 注释（含参数和返回值说明）？
- [ ] API 接口是否有 Swagger/OpenAPI 文档注释？
- [ ] 魔法数字/字符串是否提取为常量并有注释？

### 错误处理
- [ ] ⚠️ 前端组件是否包含错误处理（try-catch 或 ErrorBoundary）？
- [ ] 异步操作是否有错误捕获（`.catch()` 或 `try/catch`）？
- [ ] 错误信息对用户友好且对开发人员有调试价值？
- [ ] 全局错误处理中间件是否正常工作（`middleware/errorHandler.js`）？

### 项目结构
- [ ] 文件放置位置是否符合项目目录结构规范？
- [ ] 新增依赖是否已在 `package.json` 中声明并安装？
- [ ] 配置文件变更是否影响其他开发者环境？

---

## 五、测试覆盖 (Testing) ⚠️

### 单元测试
- [ ] ⚠️ 新增/修改的函数是否有对应的单元测试？
- [ ] 测试用例是否覆盖了正常路径和异常路径？
- [ ] Mock 数据是否合理且具有代表性？
- [ ] 测试是否独立可运行，不依赖外部状态？

### 测试覆盖率
- [ ] ⚠️ 测试覆盖率不能低于主分支当前水平
- [ ] 关键业务逻辑覆盖率是否达到 80% 以上？
- [ ] 分支覆盖率是否覆盖了所有条件分支？

### 集成与边界测试
- [ ] API 接口是否有集成测试（参考 `tests/` 目录下的测试模式）？
- [ ] 边界值测试：0、负数、最大值、空值、超长字符串
- [ ] 并发场景测试（如适用）
- [ ] 数据一致性测试（如订单状态流转）

---

## 六、提交规范 (Commit Standards)

### Commit Message
- [ ] 格式: `<type>(<scope>): <subject>`
- [ ] 类型: `feat` / `fix` / `docs` / `refactor` / `chore` / `style` / `test`
- [ ] subject 简明扼要，不超过 50 个字符
- [ ] body（如有）详细描述变更原因和内容

### 分支策略
- [ ] 功能开发基于 `feature/*` 分支
- [ ] 紧急修复基于 `hotfix/*` 分支
- [ ] 合并目标分支是否正确（develop / main）
- [ ] PR/MR 描述是否清晰，包含测试步骤
